## Who Generates What (Authoritative Flow)

This section states unambiguously what each agent produces and how execution and scoring fit in.

| Agent | Generates / Does | Output form |
|-------|------------------|-------------|
| **ObservationAgent** | UI elements, page structure, (with browser-use) flow_steps and pages | `ui_elements`, `page_context`, `flow_steps`, `pages` |
| **RequirementsAgent** | **Complete test case in BDD form** — i.e. the full scenario (Given/When/Then) that describes *what* to test | BDD **scenarios** (Given/When/Then), test_data, coverage_metrics |
| **AnalysisAgent** | Converts those BDD scenarios to executable steps, **runs test execution** (Phase 2 engine), and **scores** using risk + execution success + other criteria | risk_scores, final_prioritization, **execution_success** (passed_steps, success_rate) |
| **EvolutionAgent** | **Improved version of test cases** — converts the same (prioritized) scenarios into the **executable test steps** that are stored and shown in the UI | Executable **test steps** (array of strings), stored in DB |

**Correct end-to-end flow:**

1. **RequirementsAgent** generates the **complete test case** (the BDD scenario). That is the “test case” as a specification: what to test, in Given/When/Then form.
2. **AnalysisAgent** takes those scenarios, converts them to steps, **runs test execution** via the Phase 2 engine, and **scores** them (risk, execution success rate, and other scoring criteria). So AnalysisAgent both executes and scores the test case.
3. **EvolutionAgent** then produces the **improved/final** version of the test cases: the concrete step list (Navigate, Click, Input, Verify) that gets stored in the database and displayed in the frontend. It uses prioritization (and optionally execution feedback) from AnalysisAgent.

So: **scenarios** come from RequirementsAgent; **execution and scoring** are done by AnalysisAgent; **stored executable steps** (the “test cases” in the UI) come from EvolutionAgent.

### Reference: Workable Test Case (Production vs UAT)

The **workable Phase 2 – 3 tiers test case** used as reference (e.g. Three HK subscription flow) has **47 steps** covering: navigate → login (in-page email/password) → plan selection → T&C → Subscribe Now → multiple Confirms → document upload → HKID/personal data → payment details → contract sign → checkout → pay.  

- **Production:** This reference flow **stops at the payment page** (no actual payment in production). A full end-to-end flow including payment and success page will be provided for **UAT** later.
- **Who generates the steps you see in the UI:** The **scenario** (what to test) is generated by **RequirementsAgent**. The **executable step list** (Navigate, Click, Input, Verify) is generated by **EvolutionAgent** from that scenario (and from prioritization/feedback from AnalysisAgent). So the test cases shown in the frontend are **not** “generated by RequirementsAgent then improved by EvolutionAgent” in a two-step refinement sense—RequirementsAgent produces the BDD scenario; EvolutionAgent produces the stored step list from the scenario (and can use observed `flow_steps` to align with the real crawl).

### What Was Wrong With Agent-Generated Steps (Before Correction)

Earlier agent-generated steps for the same goal (e.g. “Subscribe to $288/month plan – complete flow”) showed these issues:

- **Wrong login path:** Steps assumed **Gmail OAuth** (new tab, Gmail sign-in, “Repeat Gmail login steps”) instead of **in-page email/password login** (click Login → email → click Login → password → click Login) as in the workable flow.
- **Placeholder credentials:** Used `testuser@example.com` because **login_credentials** were not passed; the workable flow uses specific test account email/password.
- **Extra/irrelevant steps:** e.g. “Download My3 App” modal, “Select” button verification—from the page but not part of the intended subscription flow.
- **Truncated flow:** Jumped from “proceed to subscription confirmation” to “Verify subscription flow is completed” and **omitted** the middle (multiple Confirms, document upload, HKID, payment form, sign, checkout, pay) that the workable 47-step case includes.

### How It Was Corrected

- **ObservationAgent** (browser-use multi-page crawl) now records **flow_steps** (ordered list of actions: navigate, click, input) and passes **pages**, **navigation_flow**, and **flow_steps** to the next agents.
- **Orchestration** passes **flow_steps** and **pages** to both **RequirementsAgent** and **EvolutionAgent**.
- **RequirementsAgent** uses **flow_steps** in the LLM prompt so the primary scenario describes the **full observed flow** (in-page login, all steps to goal) and is tagged e.g. `end-to-end` / `user-requirement`.
- **EvolutionAgent** for that end-to-end scenario **builds the step list from flow_steps** (so steps match the crawl: in-page login, no Gmail). When **login_credentials** are provided (e.g. via API), email/password steps use them; otherwise placeholders like `[user's email]` are used.
- **Production vs UAT:** For production, the crawl (and thus flow_steps) naturally stops at payment; when a full UAT flow is provided later, running the pipeline on UAT will produce steps that follow that environment’s flow.

**Gmail / in-page login fix (workable test case alignment):**
- **ObservationAgent** now instructs the browser-use agent to use **only in-page email/password login**: “Do NOT click Login with Gmail or any social/OAuth login. Do NOT open new tabs for login.” Gmail/OTP handling is **conditional**: it is only added when the user instruction mentions OTP, “verify email”, or “gmail”, or when `require_otp: true` is passed in the payload. For typical purchase-flow instructions, the crawl follows the same path as the workable 47-step case (in-page login only).
- **RequirementsAgent** prompt states that the end-to-end scenario must use in-page email/password login only, and must not include Gmail/OAuth unless the user explicitly requires OTP or email verification.
- **EvolutionAgent** prompt states: “Use ONLY in-page email/password login” and “Do NOT generate steps for Gmail, OAuth, new-tab login.” So even when the LLM path is used (e.g. no flow_steps), generated steps stay in-page and match the model workable test case.

**Credentials in test steps (same as Phase 2 – 3 tier):**
- Phase 2 test cases and execution can have login credentials **in the step text** (e.g. “Input 'pmo.andrewchan+010@gmail.com' into email field”, “Input 'cA8mn49&' into password field”). The agent does the same when **login_credentials** are provided:
  - **flow_steps path:** `_flow_steps_to_test_steps` already embeds the actual email and password from `login_credentials` into the step strings.
  - **LLM path:** The prompt now includes the full email and password and instructs the LLM to embed them literally in the generated steps so the test case is executable as-is (same as Phase 2). Example steps in the prompt use the real values.
- So when the API request includes `login_credentials`, the stored test case will contain those credentials in the step text, like the workable 47-step model test case. If `login_credentials` are not provided, steps use placeholders (e.g. `[user's email]`) so the flow is still correct but values must be filled in later.

## 🔄 How the 4 Agents Work

### Overview: Sequential Pipeline

```
User Input (URL + Instruction)
    ↓
[1] ObservationAgent → UI Elements + Page Structure (+ flow_steps if browser-use)
    ↓
[2] RequirementsAgent → Complete test case in BDD form (scenarios: Given/When/Then)
    ↓
[3] AnalysisAgent → Converts scenarios to steps, runs test execution, scores (risk + execution success + other criteria) → risk_scores, prioritization, execution_success
    ↓
[4] EvolutionAgent → Improved executable test steps (from prioritization + optional feedback) → stored in DB
    ↓
Output: Test Cases in Database (step list shown in UI)
```

## 📋 Agent 1: ObservationAgent

### Purpose
**Observes and extracts UI elements from web pages using Playwright + LLM.**

### What It Does

1. **Crawls Web Pages:**
   - Uses Playwright to navigate to URL
   - Follows links up to `max_depth` (currently: random link following)
   - Extracts page structure, title, links

2. **Extracts UI Elements:**
   - **Buttons:** All `<button>`, `<input type="button">`, `<input type="submit">`
   - **Inputs:** All `<input>`, `<textarea>`
   - **Links:** All `<a href>`
   - **Forms:** All `<form>` with fields

3. **LLM Enhancement:**
   - Sends page HTML to Azure OpenAI
   - LLM finds elements Playwright might miss
   - Merges Playwright + LLM results

## 📋 Agent 3: AnalysisAgent

### Purpose
**Analyzes scenarios for risk, prioritization, and executes critical scenarios in real-time.**

### What It Does

1. **Risk Scoring (FMEA Framework):**
   - **Severity:** Impact if scenario fails (1-10)
   - **Occurrence:** Likelihood of failure (1-10)
   - **Detection:** Ease of detection (1-10)
   - **RPN (Risk Priority Number):** Severity × Occurrence × Detection

2. **Business Value Scoring:**
   - User impact
   - Business criticality
   - Revenue impact

3. **ROI Calculation:**
   - Cost of testing vs. cost of failure
   - Prioritizes high-ROI scenarios

4. **Real-Time Execution:**
   - Executes scenarios with RPN ≥ 80 automatically
   - Uses 3-Tier Execution Engine (Playwright → Hybrid → Stagehand AI)
   - Measures actual success rates

5. **Prioritization:**
   - Combines risk scores, business value, ROI
   - Generates final prioritized list

### Input
```python
TaskContext(
    task_type="risk_analysis",
    payload={
        "scenarios": [...],  # From RequirementsAgent
```
