ReqIQ + Hermes Agent Agentic QA System

A fully self-hosted, multi-agent test automation architecture that converts fragmented requirement documents (PPTX, Figma, Email, Word) into executable test cases, runs them on remote Windows browser PCs, and delivers results via Telegram — with zero cloud dependency.

The 5 Agents — Roles & Responsibilities

Each agent has exactly one job. This single-responsibility design makes the system debuggable — if something fails, you know exactly which agent to look at.

🧭

Manager Agent

Hermes Agent — Claude 3.5 Sonnet via OpenRouter

The single entry point for all humans. Receives Telegram messages from developers and UAT team. Understands natural language intent and routes to the correct specialist. Never executes tasks directly — it only delegates. Holds the "big picture" memory of what tests exist, which features are covered, and what was last run.

Always-on Telegram Gateway delegate_task
📋

Requirements Agent

Hermes Profile — GPT-4o (multi-modal for Figma/PPTX screenshots)

The missing piece in your current setup. Accepts a bundle of requirement sources — any combination of Figma URLs, uploaded PPTX, email text, Word docs, or nothing at all. Calls ReqIQ's compilation API to parse and vectorise documents. Scores completeness (0–100). If below 60%, asks specific clarifying questions before proceeding. Outputs a standardised Test Instruction JSON.

Calls ReqIQ API Figma scraping JSON output
🏭

Test Case Generation Agent

Your Existing Website API — BrowserUse crawl mode

Receives the Test Instruction JSON from the Requirements Agent and calls your existing test automation website's generation API. BrowserUse crawls the target feature to discover actual UI elements. Returns a structured list of test cases with steps, expected outcomes, and selector hints. Stores results back to ReqIQ's feature registry via REST.

POST /api/generate BrowserUse crawl ReqIQ storage
🔀

Dispatcher Agent

Hermes Profile — Local Qwen 2.5:7b via Ollama (zero cost)

Routes test batches to available Windows PCs. Queries each PC's health endpoint to find the least-loaded machine. Splits large test suites across multiple PCs for parallel execution. SSHes into Windows via PowerShell or calls your execution REST API. Monitors run status with polling. Collects result JSON and screenshot paths when complete.

Load balancing SSH / REST Parallel dispatch
📊

Reporter Agent

Hermes Profile — GPT-4o-mini (cheap, templated output)

Aggregates raw results from all Windows PCs. Compares against previous run baselines to detect regressions vs known failures. Generates a structured Telegram report with pass/fail counts, failure screenshots, and plain-English failure summaries for the UAT team. Escalates critical failures via @mention. Archives full report as PDF to Garage S3.

Telegram report Regression diff PDF archive
📚

ReqIQ — Not an Agent, but the Brain

FastAPI webapp — always running alongside Hermes

ReqIQ is the knowledge backbone — not an agent itself, but the shared intelligence layer all agents query. It maintains the vector database of parsed requirements, the feature registry, the test case library, and the compilation pipeline. Agents call its REST API; humans can also use its web UI directly to upload documents and review compiled test instructions.

REST API :8000 ChromaDB RAG Web UI :3000

End-to-End Flows

Three distinct trigger paths, each producing the same outcome: test cases generated and executed without developer involvement.

🆕 Flow A — New Feature Test (Developer or UAT-triggered)

1
Human uploads requirement documents to Telegram
Developer or UAT team member sends message to Hermes bot: "New checkout flow — attaching Figma link, URS doc, and email thread from PM." Can be any combination — or nothing at all.
# Example Telegram message
"@hermesbot test new checkout flow
figma: https://figma.com/file/xxx
[attaches: checkout_URS_v2.docx]
[attaches: pm_email_thread.eml]"
2
Manager Agent routes to Requirements Agent
Hermes Manager identifies intent as "generate tests from requirements" and delegates to the Requirements Agent profile with all attached files.
3
Requirements Agent calls ReqIQ compilation API
Uploads all documents to ReqIQ. Triggers the 5-stage compilation pipeline: Parse → Extract → Merge → Score → Export. ReqIQ uses Ollama locally — no external LLM calls.
# ReqIQ API call
POST http://localhost:8000/api/v1/compile
{
"feature_name": "checkout_flow",
"documents": ["doc_id_1", "doc_id_2"],
"figma_url": "https://figma.com/file/xxx"
}
4
Completeness score returned — gap check
ReqIQ returns a completeness score. If ≥60%, proceed automatically. If <60%, Requirements Agent replies to the human with specific questions ("The document doesn't specify error handling for invalid card numbers — should the test expect a toast or inline error?").
5
Test Case Generation Agent creates test cases
Receives Test Instruction JSON from ReqIQ. Calls your existing test automation website's generation API. BrowserUse crawls the live feature to discover UI elements and augment the test steps.
# Your existing API call
POST https://your-test-site/api/generate
{
"instruction": { /* Test Instruction JSON from ReqIQ */ },
"crawl_url": "https://staging.yourapp.com/checkout",
"mode": "browseruse_crawl"
}
6
Dispatcher Agent sends to Windows PC
Checks PC health endpoints. Routes test batch to least-loaded Windows PC. Calls your execution API (or SSHes to trigger the runner directly).
7
Reporter Agent delivers results to Telegram
Aggregates results. Formats human-readable report. Attaches failure screenshots. Sends to the original requester. UAT team gets plain-English pass/fail; developer gets full technical detail.
# Telegram report (UAT-friendly version)
✅ Checkout Flow — 14/16 tests passed
❌ FAIL: Payment declined screen — expected toast, got redirect
❌ FAIL: Apple Pay button missing on mobile viewport
[📸 screenshot_fail_01.png]
[📸 screenshot_fail_02.png]
Full report: http://reqiq.local/reports/run_291

⏰ Flow B — Scheduled Nightly Regression (No Human Required)

# Cron job on Ubuntu mini PC
0 2 * * * hermes "run full regression suite on staging"

# Hermes wakes up, checks ReqIQ feature registry for all active test suites
# Dispatcher routes all suites across available Windows PCs in parallel
# Reporter compares against last night's baseline, flags regressions only
# Sends summary to #qa-channel on Telegram at ~6am before team arrives

🚀 Flow C — Post-Deploy Smoke Test (CI/CD Hook)

# From your CI/CD pipeline (GitHub Actions, Jenkins, etc.)
curl -X POST http://hermesminipc.local:8080/api/trigger \
  -d '{"trigger": "smoke_test", "env": "staging", "build": "v2.4.1"}'

# Hermes runs only smoke test suite (fastest ~5 min subset)
# Returns pass/fail webhook to CI/CD — can gate deployment on failure
# Notifies dev who triggered the deploy directly via Telegram DM

ReqIQ ↔ Hermes Integration Points

Five REST API contracts between Hermes agents and ReqIQ. Add these endpoints when building ReqIQ in Cursor AI.

Endpoint Called By Purpose Key Response Fields
POST /api/v1/documents/upload Requirements Agent Upload PPTX, DOCX, PDF, or email text for a feature. Triggers async parsing pipeline. doc_id, job_id, status
POST /api/v1/compile/{feature_id} Requirements Agent Run 5-stage compilation: parse → extract → RAG merge → score → export Test Instruction JSON completeness_score, gaps[], test_instruction_json
GET /api/v1/features Manager Agent, Dispatcher List all features in the registry with last-run status, test count, coverage score features[], total_tests, last_run_at
POST /api/v1/runs Dispatcher Agent Log a test run result — links run ID to feature, stores pass/fail, screenshots, duration run_id, regression_count, new_failures[]
GET /api/v1/runs/{run_id}/report Reporter Agent Fetch structured report data for Telegram formatting and PDF generation summary, failures[], screenshots[], pdf_url

Hermes MCP Tool Configuration

Each API above is registered as a Hermes MCP tool in ~/.hermes/tools/reqiq.yaml. Agents call them by name: reqiq_upload_document, reqiq_compile_feature, reqiq_list_features, reqiq_log_run, reqiq_get_report.

What to Add to Your Existing Test Website

Your current website already has test case generation (BrowserUse crawl) and test execution (Stagehand + Playwright). Add these three API endpoints to complete the integration:

➕

POST /api/execute

Already exists — just document it

Expose your existing Stagehand/Playwright runner as a REST endpoint accepting the Test Case JSON from ReqIQ. Returns run_id, status, result_url.

🔍

GET /api/status/{run_id}

New — needed for Dispatcher polling

Dispatcher Agent polls this every 15 seconds to check if a run is still in-progress, completed, or failed. Returns status, progress percentage, partial results.

❤️

GET /api/health

New — needed for load balancing

Returns machine load: active_runs, cpu_percent, available_slots. Dispatcher queries this on all Windows PCs before routing to determine least-loaded target.

Key Design Decisions

Why the architecture is shaped this way — and the trade-offs involved.

Why non-headless browser on Windows (not Ubuntu)?

Windows PCs handle NTLM authentication, ActiveX legacy components, and Windows-specific rendering that Linux headless Chrome cannot replicate. Screenshots of actual browser windows also provide unambiguous visual evidence for UAT teams. The Ubuntu mini PC runs zero browser processes — it only orchestrates.

Why local Ollama for ReqIQ, not OpenRouter?

Requirement documents contain confidential business logic, UX specs, and unreleased feature details. Running all compilation locally in Ollama means this data never leaves the LAN. OpenRouter is reserved for the Manager Agent (routing decisions, low sensitivity) and Requirements Agent (multi-modal understanding with less sensitive context).

Why 5 separate agents instead of one "super-agent"?

Single-agent systems fail silently and are impossible to debug. When the Dispatcher fails to route correctly, you know exactly which agent and which prompt to fix. Each agent also uses the cheapest model appropriate for its task — the free Qwen handles dispatch, not GPT-4o. This reduces LLM cost by ~70% vs a single high-capability agent.

Why ReqIQ as a separate webapp vs inside Hermes?

ReqIQ has a human-facing web UI for uploading documents, reviewing compiled instructions, and editing test cases directly. Embedding it inside Hermes would make it unusable without Telegram. The separation also lets you build ReqIQ independently in Cursor AI and integrate it into Hermes progressively.

Docker Compose — Same-Machine Deployment

All services on the Ubuntu mini PC start with one command. Hermes runs as a systemd service outside Docker for direct hardware access.

# docker-compose.yml — Ubuntu Mini PC
version: '3.9'
services:

  reqiq-api:
    build: ./reqiq
    ports: ["8000:8000"]
    volumes: ["./data/chromadb:/app/chromadb", "./data/sqlite:/app/db"]
    environment: {OLLAMA_HOST: "http://ollama:11434"}
    depends_on: [ollama, redis]

  reqiq-worker: # ARQ task queue worker
    build: ./reqiq
    command: arq app.worker.WorkerSettings
    depends_on: [reqiq-api, redis]

  reqiq-ui:
    build: ./reqiq/frontend
    ports: ["3000:3000"]

  ollama:
    image: ollama/ollama:latest
    ports: ["11434:11434"]
    volumes: ["ollama_data:/root/.ollama"]
    deploy: {resources: {limits: {memory: "8g"}}}

  garage: # Self-hosted S3
    image: dxflrs/garage:v1.0
    ports: ["3900:3900"]
    volumes: ["garage_data:/var/lib/garage"]

  redis:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb

volumes: {ollama_data: {}, garage_data: {}}

# Hermes Agent runs as systemd service (outside Docker)
# sudo systemctl enable hermes && sudo systemctl start hermes

RAM Budget on Ubuntu Mini PC

Ollama (qwen2.5:7b): ~5GB VRAM/RAM · ReqIQ API + Worker: ~400MB · ChromaDB: ~200MB · Redis: ~256MB · Garage S3: ~128MB · Total: ~6–7GB. A mini PC with 16GB RAM is comfortable; 12GB is minimum. Hermes Agent itself uses ~200MB additional.

Recommended Build Order

Build and validate each phase before starting the next. Every phase is independently testable.

1

Phase 1 — Wire Hermes to Your Existing APIs

1–2 days

Register /api/execute, /api/status, and /api/health on your test website. Configure Hermes MCP tool YAML. Test: send "run smoke test" on Telegram and get results back. Validate the execution pipeline works end-to-end before adding requirement intelligence.

2

Phase 2 — Build ReqIQ Core

1–2 weeks in Cursor AI

Build SQLModel data models, document upload endpoint, and the 5-stage Ollama compilation pipeline. Test: upload a PPTX and get back a Test Instruction JSON. No agent integration yet — test ReqIQ as a standalone tool first.

3

Phase 3 — Requirements Agent Profile

2–3 days

Create the Requirements Agent Hermes profile. Register ReqIQ REST endpoints as MCP tools. Test full flow: "generate tests for checkout feature" → Telegram → Requirements Agent → ReqIQ → Test Instruction JSON → your existing generation API → results.

4

Phase 4 — Multi-PC Dispatcher

2–3 days

Add /api/health endpoint to Windows test website. Build Dispatcher Agent profile with SSH/REST routing logic. Add second Windows PC. Test parallel execution of independent test suites.

5

Phase 5 — Reporter + Scheduled Runs

1–2 days

Build Reporter Agent. Configure cron job for nightly regression. Set up baseline comparison (first run = baseline, subsequent = diff). Validate UAT team can understand the Telegram report without IT help.

6

Phase 6 — Hardening

Ongoing

Add Tailscale for remote access. Enable ReqIQ completeness scoring and gap-asking behaviour. Build PDF report archiving. Add CI/CD webhook for post-deploy smoke tests. Monitor RAM usage on mini PC.
